# Discord Announcement Network Relay
## Cursor Implementation Plan

## 1. Objective

Build a Discord bot that runs in one central Discord server and relays followed announcement messages through one or more network announcement channels.

The bot must support this workflow:

```text
External server announcement channel
    -> Discord Channel Follow
    -> Dedicated source channel inside a feed category
    -> Bot transforms the message
    -> Optional concat/audit channel
    -> Network announcement channel
    -> Bot publishes the message
    -> Other servers receive it through Discord Channel Following
```

Example:

```text
External Server A #announcements
    -> Central Server / Stingers Feed / #server-a
    -> Bot
    -> [Original Author] [Server A Emoji]
       Original message
    -> Central Server / The Network / #stingers
    -> Published announcement
```

The bot is installed only in the central Discord server.

---

## 2. Core Requirements

### 2.1 Feed routing

Each network has:

- One output announcement channel.
- One feed category.
- Zero or more source channels inside that feed category.
- An optional concat channel inside the feed category.
- One profile forum containing a thread for each participating server.

Example:

```text
The Network
  #stingers

Stingers Feed
  #concat
  #server-a
  #server-b
  #server-c

Profiles
  Server Profiles forum
    Server A profile thread
    Server B profile thread
    Server C profile thread
```

Messages received in any configured source channel must be routed to the output announcement channel associated with that feed category.

The bot must ignore:

- Messages outside configured feed categories.
- Messages posted in the concat channel.
- Messages sent by the relay bot itself.
- Messages that are not webhook-delivered followed announcements, unless an administrator explicitly enables manual-message relay.
- Message updates generated by Discord after a message has already been relayed.

---

### 2.2 Profile system

Profiles should be represented by forum threads.

Each profile thread corresponds to one participating server.

The profile thread must define:

- Server display name.
- Source feed channel.
- Enabled or disabled state.
- Optional network override.
- Optional custom display label.
- Profile image.

The first valid image attached to the forum starter message is the profile image.

Recommended profile body format:

```yaml
server_name: Vanguard
source_channel: 123456789012345678
enabled: true
network: stingers
display_name: Vanguard
```

The bot should accept either:

- A raw channel ID.
- A channel mention such as `<#123456789012345678>`.

The forum thread title may be used as a fallback server name, but Discord IDs must be used as the persistent identity.

---

### 2.3 Emoji generation

For each profile image, the bot must create one server-specific custom emoji in the central guild.

Emoji behavior:

1. Download the first valid image from the profile thread starter message.
2. Convert it to PNG.
3. Resize it to fit Discord emoji limits.
4. Calculate a SHA-256 hash of the normalized image.
5. Compare the hash with the stored profile image hash.
6. If unchanged, do nothing.
7. If changed:
   - Create the replacement emoji first.
   - Save the new emoji ID.
   - Delete the old emoji only after the new emoji succeeds.
8. If the profile is deleted or disabled, retain the emoji by default.
9. Provide an administrative cleanup command for unused emojis.

Generated emoji names should be deterministic:

```text
net_<slug>_<short_channel_id>
```

Example:

```text
net_vanguard_345678
```

The emoji ID, not the emoji name, must be used when rendering messages.

If emoji creation fails because the guild is at its emoji limit:

- Do not block the profile from existing.
- Mark the profile as degraded.
- Use a fallback symbol such as `◈`.
- Log the error in the relay log channel.

---

### 2.4 Message transformation

Input:

```text
Raid starts at 8 PM.
```

Output:

```text
**Original Username** <:net_vanguard:123456789012345678>

Raid starts at 8 PM.
```

Required header format:

```text
[username] [server-specific emoji]
```

Recommended Discord formatting:

```text
**Username** <:emoji_name:emoji_id>
```

The original webhook author name should be used for `username`.

The source server identity must come from the profile mapped to the source channel, not from the webhook author name.

The bot must preserve, where possible:

- Message text.
- Attachments.
- Embeds.
- URLs.
- Basic markdown.
- Original author name.
- Original source profile emoji.

The bot must suppress mentions:

```python
discord.AllowedMentions.none()
```

The bot must not relay `@everyone`, `@here`, user mentions, or role mentions as live mentions.

For unsupported content:

- Stickers: append the sticker name and URL when available.
- Polls: append a plain-text summary when available.
- Components/buttons: do not recreate interactive components.
- Replies: optionally include a short quoted reply reference.
- Empty messages: relay only if attachments, embeds, stickers, or other supported content exist.

---

### 2.5 Message length handling

Discord messages have a content length limit.

The relay must not silently truncate without context.

Preferred behavior:

1. Construct the transformed message.
2. If it fits, send normally.
3. If it exceeds the limit:
   - Send the header in the first message.
   - Split content into continuation messages at paragraph or newline boundaries.
   - Publish each continuation message.
   - Prefix continuation messages with:
     ```text
     ↳ continued
     ```
4. Keep attachments on the first message unless Discord upload constraints require otherwise.

All continuation messages should be linked in the relay database to the same source message.

---

### 2.6 Concat channel behavior

The concat channel is optional.

Support two relay modes:

#### Direct mode

```text
Source feed channel
    -> Bot
    -> Output announcement channel
    -> Publish
```

#### Concat mode

```text
Source feed channel
    -> Bot
    -> Concat channel
    -> Bot
    -> Output announcement channel
    -> Publish
```

Recommended initial implementation:

- Use direct mode for production relaying.
- Mirror a plain-text audit copy to `#concat`.
- Do not use `#concat` as a second event-driven relay hop.
- This avoids loops, duplicate messages, and unnecessary complexity.

The concat channel should function as an audit stream showing exactly what was sent.

---

### 2.7 Publishing

After sending a recreated message into the configured output announcement channel, the bot must publish it.

Requirements:

- Verify the destination is an announcement channel.
- Send the recreated message.
- Call the publish/crosspost action.
- Retry transient failures.
- Do not retry permanent permission failures indefinitely.
- Log publish failures.

The bot needs appropriate permissions in the output channel, including permission to send messages and publish announcements.

## 6. Database Schema

### 6.1 `networks`

```sql
CREATE TABLE networks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    guild_id INTEGER NOT NULL,
    key TEXT NOT NULL UNIQUE,
    display_name TEXT NOT NULL,
    feed_category_id INTEGER NOT NULL UNIQUE,
    output_channel_id INTEGER NOT NULL UNIQUE,
    concat_channel_id INTEGER,
    enabled INTEGER NOT NULL DEFAULT 1,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

### 6.2 `profiles`

```sql
CREATE TABLE profiles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    guild_id INTEGER NOT NULL,
    profile_thread_id INTEGER NOT NULL UNIQUE,
    profile_starter_message_id INTEGER NOT NULL UNIQUE,
    source_channel_id INTEGER NOT NULL UNIQUE,
    network_id INTEGER NOT NULL,
    server_name TEXT NOT NULL,
    display_name TEXT NOT NULL,
    enabled INTEGER NOT NULL DEFAULT 1,
    emoji_id INTEGER,
    emoji_name TEXT,
    image_hash TEXT,
    image_source_url TEXT,
    degraded_reason TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    FOREIGN KEY (network_id) REFERENCES networks(id)
);
```

### 6.3 `relay_records`

```sql
CREATE TABLE relay_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_message_id INTEGER NOT NULL UNIQUE,
    source_channel_id INTEGER NOT NULL,
    source_webhook_id INTEGER,
    profile_id INTEGER NOT NULL,
    network_id INTEGER NOT NULL,
    destination_channel_id INTEGER NOT NULL,
    destination_message_ids TEXT NOT NULL,
    status TEXT NOT NULL,
    error_message TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    FOREIGN KEY (profile_id) REFERENCES profiles(id),
    FOREIGN KEY (network_id) REFERENCES networks(id)
);
```

`destination_message_ids` may initially be stored as a JSON array.

### 6.4 `settings`

```sql
CREATE TABLE settings (
    key TEXT PRIMARY KEY,

    updated_at TEXT NOT NULL
```

## 8. Discord Event Handling

## 8.1 Startup

On startup:

1. Connect to Discord.
2. Initialize the database.
3. Run migrations.
4. Validate the configured guild.
5. Load networks and profiles into caches.
6. Sync application commands to the central guild.
7. Validate channel permissions.
8. Log startup status.

Do not automatically rebuild every emoji on each startup.

Only repair missing emoji records when explicitly requested or when profile validation detects a missing emoji.

---

## 8.2 Incoming feed messages

Handle `on_message`.

Processing order:

```text
1. Is the message in the configured guild?
2. Is the message in a configured source channel?
3. Is the profile enabled?
4. Is the network enabled?
5. Is the message already present in relay_records?
6. Is the message from the relay bot?
7. Is it a webhook message?
8. If manual relay is disabled and webhook_id is null, ignore it.
9. Transform the message.
10. Send it to the output announcement channel.
11. Publish it.
12. Write an audit message to concat, if configured.
13. Persist relay_records.
```

Use a per-source-message lock to prevent duplicate processing.

Example lock key:

```text
relay:<source_message_id>
```

An in-memory `asyncio.Lock` map is sufficient for one bot process.

---

## 8.3 Profile forum events

Handle:

- `on_thread_create`
- `on_thread_update`
- `on_message`
- `on_message_edit`
- `on_raw_message_delete`
- `on_raw_thread_delete`

Profile synchronization should run when:

- A new forum thread is created.
- The starter message is edited.
- The thread title changes.
- An attachment is added or replaced.
- An administrator runs `/profile sync`.

Because Discord event behavior can vary for forum starter messages, provide a manual sync command as the authoritative recovery mechanism.

---

## 8.4 Source message edits

Version 1 behavior:

- Do not automatically edit already published relay messages.
- Log that the original message was edited.
- Provide `/relay refresh <message_link>` later if edit synchronization is desired.

This avoids complex reconciliation across multiple published continuation messages.

Version 2 may support edit propagation.

---

## 8.5 Source message deletion

Version 1 behavior:

- Do not automatically delete published announcements.
- Log the source deletion.
- Keep the relay record.

Version 2 may support deletion propagation behind a configuration flag.

## 10. Profile Parser

The parser must be tolerant but deterministic.

Supported input:

```yaml
server_name: Vanguard
source_channel: <#123456789012345678>
enabled: true
network: stingers
display_name: Vanguard
```

Parsing rules:

- Ignore blank lines.
- Ignore lines starting with `#`.
- Keys are case-insensitive.
- Trim whitespace around keys and values.
- Reject duplicate keys.
- Validate booleans.
- Validate Discord IDs.
- Fall back to the thread title for `server_name`.
- Fall back to `server_name` for `display_name`.
- Resolve `network` by key.
- If `network` is absent, infer it from the source channel parent category.

If the profile cannot be parsed:

- Do not overwrite a previously valid profile.
- Mark the sync as failed.
- Reply in the forum thread with a concise error.
- Log the full error in the relay log channel.

---

## 11. Image Processing

Use Pillow.

Normalization pipeline:

```text
Input image
    -> Verify supported format
    -> Convert to RGBA
    -> Center-crop to square
    -> Resize to 128x128
    -> Save optimized PNG
    -> Verify byte size
    -> Reduce dimensions or optimize further if necessary
    -> Compute SHA-256
```

Supported formats:

- PNG
- JPEG
- WEBP
- GIF

For animated GIFs in version 1:

- Use the first frame.
- Convert it to static PNG.

Reject:

- SVG.
- Files that cannot be decoded safely.
- Excessively large files after download.
- Non-image attachments.

Use a maximum download size.

Do not trust file extensions alone.

## 12. Message Formatting Service

Expose a pure function where possible:

```python
def build_relay_content(
    author_name: str,
    profile: ServerProfile,
    source_content: str,
) -> str:
    ...
```

Header:

```python
emoji = (
    f"<:{profile.emoji_name}:{profile.emoji_id}>"
    if profile.emoji_id and profile.emoji_name
    else "◈"
)

header = f"**{sanitize_author(author_name)}** {emoji}"
```

Final body:

```python
return f"{header}\n\n{source_content}".strip()
```

Sanitize the author name so it cannot inject unwanted mentions.

Do not strip useful markdown from the source content.

## 13. Attachment Handling

For each attachment:

1. Validate file size.
2. Download with a timeout.
3. Re-upload to the destination message.
4. Preserve the original filename when safe.
5. Log skipped attachments.

If the combined attachment size exceeds Discord limits:

- Send attachments across multiple continuation messages.
- Preserve order.
- Add an audit warning.

Do not persist attachment bytes to disk longer than necessary.

Use in-memory buffers where practical.

---

## 14. Embed Handling

Attempt to reuse source embeds when Discord permits it.

Rules:

- Copy up to Discord's embed limit.
- Remove unsafe or unsupported fields only when required.
- Preserve image and thumbnail URLs.
- Preserve titles, descriptions, fields, and footer text.
- Do not recreate interactive components.
- If an embed cannot be sent, fall back to a plain-text summary.

Avoid copying Discord-generated embeds that merely duplicate a URL already present in the message when doing so would create visible duplication.

---

## 15. Deduplication

Use the source Discord message ID as the primary idempotency key.

Before relaying:

```sql
SELECT 1
FROM relay_records
WHERE source_message_id = ?;
```

If a record exists:

- Ignore the event.
- Do not resend.
- Log only at debug level.

Create the relay record in a pending state before sending.

State flow:

```text
pending
    -> sent
    -> published
```

Failure states:

```text
failed_send
failed_publish
partial
```

This allows retry commands to recover incomplete relays.

---

## 16. Error Handling

Create explicit exception types:

```python
class RelayError(Exception): ...
class ProfileParseError(Exception): ...
class EmojiSyncError(Exception): ...
class RoutingError(Exception): ...
class PermissionValidationError(Exception): ...
```

Retry only transient errors:

- Rate limits.
- Temporary network failures.
- Discord server errors.

Do not retry:

- Missing permissions.
- Invalid channel type.
- Deleted channel.
- Invalid profile.
- Emoji capacity reached.

Use exponential backoff with jitter.

Maximum automatic retries:

```text
3
```

## 18. Permissions Validation

At startup and through `/maintenance permissions`, validate:

### Profile forum

- View Channel.
- Read Message History.
- Send Messages in Threads.
- Manage Threads, if the bot posts status responses.

### Feed source channels

- View Channel.
- Read Message History.

### Concat channel

- View Channel.
- Send Messages.
- Embed Links.
- Attach Files.

### Output announcement channel

- View Channel.
- Send Messages.
- Embed Links.
- Attach Files.
- Publish Messages or the effective Discord permission required by the library/API.

### Guild-level emoji management

- Create Expressions or equivalent current permission.
- Manage Expressions or equivalent current permission.

The implementation should use the permission names exposed by the installed `discord.py` version rather than hardcoding assumptions.

## 19. Security Requirements

- Store the bot token only in environment variables.
- Never log the bot token.
- Disable live mentions in relayed messages.
- Restrict slash commands to administrators.
- Validate all Discord IDs against the central guild.
- Limit attachment download size.
- Use timeouts for all downloads.
- Do not execute content from profile messages.
- Do not accept arbitrary URLs for profile images in version 1.
- Use only Discord-hosted attachments for profile images.
- Escape or sanitize user-controlled text used in logs.
- Do not expose stack traces in public channels.

## 21. Acceptance Criteria

The first release is complete when all of the following work:

1. An external announcement is published.
2. Discord delivers it to a dedicated source channel.
3. The bot recognizes the source channel.
4. The bot loads the correct server profile.
5. The bot formats:
   ```text
   **Original Username** <server emoji>

   Original content
   ```
6. Attachments are copied.
7. Supported embeds are copied.
8. Live mentions are suppressed.
9. The recreated message is sent to the output announcement channel.
10. The recreated message is published.
11. A receiving server obtains the published announcement.
12. Duplicate source events do not create duplicate messages.
13. Updating the profile image creates a new emoji.
14. The old emoji is deleted only after successful replacement.
15. Invalid profiles do not interrupt other relays.
16. Relay failures are logged and can be retried.

## 27. Important Architectural Decisions

1. Source channel ID is the authoritative server identity.
2. Forum profiles are configuration records, not free-form discussion threads.
3. Network routing is based on feed category.
4. Concat is an audit mirror, not a second relay hop.
5. Messages are recreated, not forwarded.
6. New messages are published from the output announcement channel.
7. Emoji replacement is transactional.
8. Source message ID provides idempotency.
9. SQLite is the initial source of truth.
10. Discord IDs are stored as integers and never inferred from names.
