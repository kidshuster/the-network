## 1. Project Concept & Objective

**The Network** (plan title: *Discord Announcement Network Relay*) is a central-server Discord bot that relays followed announcement messages through one or more network announcement channels.

The bot is installed **only in the central Discord server**. External servers publish announcements that Discord delivers via **Channel Follow** into dedicated source channels inside a feed category. The bot transforms those messages, optionally mirrors them to an audit concat channel, sends them to a network output announcement channel, **publishes** them, and downstream servers receive them through Discord Channel Following.

### Relay workflow

```text
External server announcement channel
    -> Discord Channel Follow
    -> Dedicated source channel inside a feed category
    -> Bot transforms the message
    -> Optional concat/audit channel
    -> Network announcement channel
    -> Bot publishes the message
    -> Other servers receive it through Discord Channel Following
```

### Example flow

```text
External Server A #announcements
    -> Central Server / Stingers Feed / #server-a
    -> Bot
    -> [Original Author] [Server A Emoji]
       Original message
    -> Central Server / The Network / #stingers
    -> Published announcement
```

## 2. Core Requirements

### 2.1 Feed routing

Each **network** has:

- One output announcement channel
- One feed category
- Zero or more source channels inside that feed category
- An optional concat channel inside the feed category
- One profile forum containing a thread for each participating server

Example layout:

```text
The Network
  #stingers

Stingers Feed
  #concat
  #server-a
  #server-b
  #server-c

Profiles
  Server Profiles forum
    Server A profile thread
    Server B profile thread
    Server C profile thread
```

Messages in any configured source channel route to the output announcement channel associated with that feed category.

**Ignore / filter rules (v1):**

- Messages outside configured feed categories
- Messages posted in the concat channel
- Messages sent by the relay bot itself (no self-relay)
- Messages that are not webhook-delivered followed announcements, unless `MANUAL_RELAY_ENABLED` is explicitly enabled
- Message updates generated by Discord after a message has already been relayed (no edit relay in v1)
- Duplicate source events (dedup via `source_message_id` in `relay_records`)

### 2.2 Profile system

Profiles are **forum threads**; each thread corresponds to one participating server.

Profile thread defines:

- Server display name
- Source feed channel
- Enabled or disabled state
- Optional network override
- Optional custom display label
- Profile image (first valid image on forum starter message)

Recommended body format (YAML-like):

```yaml
server_name: Vanguard
source_channel: 123456789012345678
enabled: true
network: stingers
display_name: Vanguard
```

Accepts raw channel ID or `<#123456789012345678>`. Thread title may fallback for server name; Discord IDs are persistent identity.

### 2.3 Emoji generation

For each profile image, create one server-specific custom emoji in the central guild:

1. Download first valid image from profile starter message
2. Convert to PNG
3. Resize to Discord emoji limits
4. SHA-256 hash normalized image
5. Compare with stored hash; if unchanged, skip
6. If changed: create replacement emoji first, save new ID, delete old emoji only after success
7. If profile deleted/disabled: retain emoji by default
8. Administrative cleanup command for unused emojis

Deterministic emoji name: `net_<slug>_<short_channel_id>` (e.g. `net_vanguard_345678`). Use emoji **ID**, not name, when rendering.

If guild emoji limit reached: do not block profile; mark degraded; fallback symbol `◈`; log to relay log channel.

### 2.4 Message transformation

Input example: `Raid starts at 8 PM.`

Output format:

```text
**Original Username** <:net_vanguard:123456789012345678>

Raid starts at 8 PM.
```

Required header: `[username] [server-specific emoji]`. Use original webhook author name for username; source server identity from profile mapped to source channel.

Preserve where possible: text, attachments, embeds, URLs, basic markdown, author name, profile emoji.

Suppress mentions via `discord.AllowedMentions.none()` — no live `@everyone`, `@here`, user, or role mentions.

Unsupported content handling:

- Stickers: append name and URL when available
- Polls: append plain-text summary when available
- Components/buttons: do not recreate
- Replies: optionally include short quoted reply reference (optional; no default chosen)
- Empty messages: relay only if attachments, embeds, stickers, or other supported content exist

### 2.5 Message length handling

Do not silently truncate.

1. Construct transformed message
2. If it fits, send normally
3. If over limit: send header in first message; split content at paragraph/newline boundaries; publish each continuation; prefix continuations with `↳ continued`
4. Keep attachments on first message unless Discord upload constraints require otherwise
5. Link all continuation messages in relay database to same source message

### 2.6 Concat channel behavior

Optional. Two conceptual modes:

- **Direct mode:** source → bot → output → publish
- **Concat mode (conceptual):** source → bot → concat → bot → output → publish

**Recommended v1:** use direct mode for production; mirror plain-text audit copy to `#concat`; do **not** use concat as a second event-driven relay hop (avoids loops and duplicates). Concat is an audit stream showing exactly what was sent.

### 2.7 Publishing

After sending recreated message to output announcement channel:

- Verify destination is an announcement channel
- Send recreated message
- Call publish/crosspost action
- Retry transient failures (max 3 automatic retries with exponential backoff and jitter)
- Do not retry permanent permission failures indefinitely
- Log publish failures

Requires send messages and publish announcements permissions in output channel.

## 3. Configuration & Data Model

### Environment variables (secrets / bootstrap only)

From `.env.example`:

| Variable | Purpose |
|----------|---------|
| `DISCORD_TOKEN` | Bot token |
| `GUILD_ID` | Central guild |
| `DATABASE_PATH` | SQLite path (default `./data/relay.db`) |
| `LOG_LEVEL` | Logging level |
| `PROFILE_FORUM_CHANNEL_ID` | Profile forum |
| `RELAY_LOG_CHANNEL_ID` | Relay log channel |
| `MANUAL_RELAY_ENABLED` | Allow non-webhook relay (default `false`) |

Network routes stored in database via slash commands — no hardcoded channel IDs in source.

### SQLite tables

**`networks`** — `key`, `display_name`, `feed_category_id`, `output_channel_id`, optional `concat_channel_id`, `enabled`, guild scoping.

**`profiles`** — forum thread ID, starter message ID, `source_channel_id` (unique), `network_id`, names, emoji ID/name, image hash/URL, `degraded_reason`, `enabled`.

**`relay_records`** — `source_message_id` (unique idempotency key), channel/webhook IDs, profile/network/destination IDs, `destination_message_ids` (JSON array), `status`, `error_message`.

**`settings`** — key/value store.

Relay state flow: `pending` → `sent` → `published`; failure states: `failed_send`, `failed_publish`, `partial`.

### Domain models

- **`Network`** — id, key, display_name, feed_category_id, output_channel_id, concat_channel_id, enabled
- **`ServerProfile`** — id, profile_thread_id, source_channel_id, network_id, server_name, display_name, enabled, emoji_id/name, image_hash, degraded_reason
- **`RelayResult`** — source_message_id, destination_message_ids, published_message_ids, success, error

### Deduplication

Primary idempotency key: **source Discord message ID**. Check `relay_records` before relaying; create record in pending state before sending. Per-source-message lock (`relay:<source_message_id>`) prevents duplicate processing.

## 4. Discord Integration Surface

### Events

**Startup:** connect, init DB, migrations, validate guild, load caches, sync slash commands, validate permissions, log status. Do not rebuild all emojis on every startup.

**`on_message` (feed):** guild check → source channel → enabled profile/network → dedup → not bot self → webhook check (unless manual relay) → transform → send → publish → concat audit → persist record.

**Profile forum:** `on_thread_create`, `on_thread_update`, `on_message`, `on_message_edit`, `on_raw_message_delete`, `on_raw_thread_delete`. Sync on thread create, starter edit, title change, attachment change, or `/profile sync`.

**Source edits (v1):** do not edit published relays; log edit; `/relay refresh` deferred.

**Source deletion (v1):** do not delete published announcements; log deletion; keep relay record.

### Slash command groups

**`/network`:** `create`, `edit`, `list`, `disable`, `enable`, `validate`, `delete`

**`/profile`:** `sync`, `sync-all`, `show`, `disable`, `enable`, `repair-emoji`, `delete`

**`/relay`:** `test`, `retry`, `status`, `recent`

**`/maintenance`:** `validate`, `cleanup-emojis`, `rebuild-cache`, `permissions`

All admin commands require `Manage Guild` or a configured moderator role.

### Permission validation (startup + `/maintenance permissions`)

- Profile forum: view, read history, send in threads, manage threads (if bot posts status)
- Feed source channels: view, read history
- Concat: view, send, embed links, attach files
- Output announcement: view, send, embed links, attach files, publish messages
- Guild: create/manage expressions (emoji)

## 7. Implementation Phases

| Phase | Focus | Deliverable |
|-------|-------|-------------|
| **1 — Bootstrap** | Project setup, discord.py, env, SQLite, migrations, logging, guild slash sync | Bot starts, connects, initializes DB, responds to `/status` |
| **2 — Network routing** | Network persistence, `/network create` & `list`, category/output validation, route cache | Bot maps feed category to output announcement channel |
| **3 — Profiles** | Forum profile parser, `/profile sync`, storage, source-channel mapping, validation | Forum thread configures one source server profile |
| **4 — Emoji sync** | Image download, Pillow normalize, hash, custom emoji create/replace, fallback | Each profile owns server-specific emoji from its image |
| **5 — Basic relay** | Feed listener, webhook filter, formatter, mention suppression, send, publish, relay records | Plain-text followed announcements relay end to end |
| **6 — Media support** | Attachments, embeds, empty-content media, text splitting, continuations | Typical Discord announcements retain useful media |
| **7 — Audit & recovery** | Concat mirror, relay log, `/relay recent` & `retry`, permission validation, cache rebuild | Administrators diagnose and recover failed relays |
| **8 — Hardening** | Full tests, Docker, rate-limit retries, cleanup commands, docs, SQLite backup | Bot ready for long-running deployment |

## 8. Acceptance Criteria

First release complete when all of the following work:

- [ ] External announcement published; Discord delivers to dedicated source channel
- [ ] Bot recognizes source channel and loads correct server profile
- [ ] Message formatted as `**Original Username** <server emoji>` plus original content
- [ ] Attachments copied
- [ ] Supported embeds copied
- [ ] Live mentions suppressed
- [ ] Recreated message sent to output announcement channel
- [ ] Recreated message published
- [ ] Receiving server obtains published announcement
- [ ] Duplicate source events do not create duplicate messages
- [ ] Profile image update creates new emoji; old emoji deleted only after successful replacement
- [ ] Invalid profiles do not interrupt other relays
- [ ] Relay failures logged and retriable

## 9. Deferred Features (Out of Scope v1)

- Automatic synchronization of source message edits
- Automatic deletion of relayed messages
- Multiple central guilds
- Web dashboard
- Arbitrary external image URLs (v1 uses Discord-hosted attachments only)
- Interactive component recreation
- Per-message moderator approval
- Cross-network message fan-out
- Per-server filtering rules
- Persistent job queues
- External database
- Horizontal scaling

## 10. Architectural Decisions

1. **Source channel ID** is authoritative server identity
2. **Forum profiles** are configuration records, not free-form discussion threads
3. **Network routing** is based on feed category
4. **Concat** is an audit mirror, not a second relay hop
5. **Messages are recreated**, not forwarded
6. **New messages are published** from the output announcement channel
7. **Emoji replacement is transactional** (create new before delete old)
8. **Source message ID** provides idempotency
9. **SQLite** is the initial source of truth
10. **Discord IDs** stored as integers; never inferred from names
